# 实时行情收盘停轮询设计

## 背景

当前页面在 [App.vue](file:///Users/fangteng/Desktop/ai-ui/src/App.vue) 中通过固定定时器执行实时刷新：

- 行情每 5 秒拉取一次；
- 趋势分析每 30 秒拉取一次。

现状没有根据 A 股交易时段动态控制轮询，因此午休、收盘后、周末等非交易时段页面仍会继续发起请求。这会带来以下问题：

- 产生无效请求，占用前后端资源；
- UI 仍显示“实时轮询中”，与实际市场状态不一致；
- 页面长时间打开时，缺少明确的时段切换控制，容易形成定时器空转或重复启动风险。

## 目标

- 按 A 股交易时段控制自动轮询，仅在连续竞价时段自动刷新。
- 午休、收盘后、周末进入休市状态时，自动停止行情与分析轮询。
- 页面如果持续打开到下一交易时段，无需刷新页面即可自动恢复轮询。
- 保留现有手动刷新能力，休市时用户仍可主动拉取一次最新快照。
- 延续现有 `AbortController + requestId` 竞态保护，避免旧响应覆盖新状态。

## 非目标

- 不修改服务端行情与分析接口协议。
- 不引入节假日交易日历、港美股时段或盘前盘后扩展逻辑。
- 不重构现有趋势分析计算逻辑与图表组件。
- 不引入新的状态管理库或定时任务库。

## 交易时段口径

本次按 A 股连续竞价时段定义“允许自动轮询”：

- 上午：`09:30:00 <= now < 11:30:00`
- 下午：`13:00:00 <= now < 15:00:00`

以下时段均视为“不允许自动轮询”：

- `00:00:00 - 09:29:59`
- `11:30:00 - 12:59:59`
- `15:00:00 - 23:59:59`
- 周六、周日全天

节假日暂不纳入本次范围。也就是说，当前版本只按“工作日 + 时分秒”判断，后续如有需要再扩展正式交易日历。

## 方案

### 前端状态模型

在 [App.vue](file:///Users/fangteng/Desktop/ai-ui/src/App.vue) 中新增显式的市场时段状态，而不是仅靠定时器是否存在来隐式判断。

建议引入以下派生状态：

- `marketPhase`：枚举当前市场阶段，如 `trading`、`lunch_break`、`pre_open`、`closed`、`weekend`
- `pollingEnabled`：当前是否允许自动轮询

这样可以让以下逻辑统一走一个入口：

- UI 状态文案展示；
- 自动轮询启停；
- 下一个时段切换点调度；
- 边界时刻的异常排查与测试。

### 交易时段工具函数

在 [App.vue](file:///Users/fangteng/Desktop/ai-ui/src/App.vue) 内新增轻量辅助函数，保持最小改动：

- `getMarketPhase(now: Date)`：返回当前所处阶段
- `isTradingSession(now: Date)`：返回当前是否允许自动轮询
- `getNextMarketBoundary(now: Date)`：返回下一个需要重新评估市场状态的时间点

其中 `getNextMarketBoundary(now)` 需要覆盖以下边界：

- 盘前返回当日 `09:30`
- 午间休市返回当日 `13:00`
- 收盘后返回下一个工作日 `09:30`
- 上午交易中返回当日 `11:30`
- 下午交易中返回当日 `15:00`
- 周末返回下一个工作日 `09:30`

这些函数保持纯函数设计，避免依赖组件状态，便于后续单测或局部自测。

### 轮询启停策略

保留现有两条自动刷新链路，但改成“按需启动、按需销毁”：

- 行情轮询：`5s`
- 分析轮询：`30s`

新增统一控制函数：

- `startPolling()`：仅在 `pollingEnabled === true` 且存在 `activeSymbol` 时启动定时器
- `stopPolling()`：清除两个轮询定时器，确保幂等，重复调用不报错
- `syncPollingByMarketPhase()`：根据当前市场状态决定调用 `startPolling()` 或 `stopPolling()`

控制原则如下：

- 页面挂载时先评估市场状态，再决定是否启动轮询
- 市场从交易中切到午休或收盘时，立即执行 `stopPolling()`
- 市场从休市切到可交易时，自动重新执行 `startPolling()`
- 切换股票、删除当前股票、空目录时，也必须复用同一套启停函数，避免散落多个定时器入口

### 时段切换调度

仅靠 `setInterval` 无法解决“页面开着过了收盘后要停、第二天开盘要恢复”的问题，因此本次新增一个独立的市场边界调度器。

建议增加：

- `marketBoundaryTimer`
- `scheduleNextMarketBoundaryCheck()`

其行为为：

1. 每次完成市场状态评估后，计算下一个边界时间；
2. 用单个 `setTimeout` 挂到该边界点附近；
3. 定时器触发后重新执行 `syncPollingByMarketPhase()`；
4. 再次递归调度下一个边界点。

这样可确保：

- 11:30 到点自动停轮询；
- 13:00 到点自动恢复轮询；
- 15:00 到点自动停轮询；
- 页面过夜后到次日 09:30 自动恢复轮询。

为避免临界点误差，可在目标边界基础上增加极小冗余，如 `+500ms` 后再重算状态。

### 请求触发口径

本次将“自动轮询”和“一次性请求”拆开处理，避免休市时页面完全没有数据。

建议口径如下：

- 首次进入页面或切换股票时：仍执行一次 `fetchQuote({ reset: true })` 与 `fetchAnalysis({ reset: true })`
- 自动轮询：仅在交易时段开启
- 手动点击“立即刷新”或分析重试按钮：始终允许触发一次请求

这样做的好处是：

- 用户在收盘后打开页面，仍能看到最新收盘快照与分析；
- “停止轮询”只影响后台自动刷新，不影响主动操作；
- 对当前页面交互改动最小。

### UI 状态文案

在 [App.vue](file:///Users/fangteng/Desktop/ai-ui/src/App.vue) 中调整顶部状态区文案，让页面反馈与真实轮询状态一致。

建议文案：

- `trading`：`实时轮询中`
- `lunch_break`：`午间休市，已停止自动刷新`
- `pre_open`：`未开盘，已停止自动刷新`
- `closed`：`已收盘，已停止自动刷新`
- `weekend`：`休市中，已停止自动刷新`

`quoteLoading` 仍然只表达“当前请求是否在执行”，不再同时承担“是否正在自动轮询”的语义，避免状态混淆。

## 状态与异常处理

### 异步与竞态控制

当前 [App.vue](file:///Users/fangteng/Desktop/ai-ui/src/App.vue) 已具备：

- `quoteRequestId` / `analysisRequestId`
- `AbortController`

本次继续沿用该机制，不改变请求主链路。重点要求：

- 自动轮询停止时，不需要强制清空现有数据
- 若恰好有请求在收盘瞬间已发出，可允许其自然返回，但不得在旧股票切换后覆盖新状态
- 新一轮自动轮询启动前，不能重复创建多个 interval

也就是说，本次重点是“控制后续调度”，而不是中断所有已发送请求。

### 空目录与切换场景

若无激活股票，则：

- `startPolling()` 必须直接返回
- 现有轮询定时器必须被清理
- 市场边界调度器可以保留，也可以在无标的时只保留状态计算，不触发请求

建议保留市场状态调度，但所有自动请求函数先判断 `activeSymbol`，无标的时直接返回。这样后续新增股票时，能自然复用同一套状态控制。

### 失败路径

- 自动轮询请求失败：沿用现有错误提示，不关闭后续交易时段内的自动轮询
- 手动刷新失败：沿用现有错误提示，允许用户再次重试
- 切到休市状态：不是错误，不显示报错，只切换状态文案
- 长时间挂机跨天：由市场边界调度器负责恢复，不要求用户手动刷新页面

## 风险与稳定性分析

本次改动的核心风险不在业务计算，而在定时器与异步时序：

- 重复启动轮询导致同一接口被多个 interval 并发触发
- 到达收盘边界时未及时清理 interval，导致休市后继续请求
- 仅停止 interval 但未安排下一次边界检查，导致次日无法自动恢复轮询
- UI 仍以 `quoteLoading` 显示“实时中”，造成状态误导
- 午休被误判成收盘，导致下午无法恢复

对应控制措施：

- `startPolling()` 内先清理旧定时器或先判断是否已存在，保证幂等
- `stopPolling()` 统一清理行情与分析两个 interval
- `scheduleNextMarketBoundaryCheck()` 只允许存在一个 `marketBoundaryTimer`
- 轮询状态文案与 loading 状态分离
- 市场阶段明确拆分为多个枚举值，午休和收盘分别建模

## 验收标准

- 工作日 `09:30-11:30`、`13:00-15:00` 自动执行实时轮询
- 工作日午休与 `15:00` 后自动停止轮询，不再继续发起自动请求
- 页面在休市时打开，默认不自动轮询，但可手动刷新查看最新快照
- 页面持续打开跨越 `11:30`、`13:00`、`15:00`、次日 `09:30` 时，轮询状态能自动切换，无需刷新页面
- 切换股票、删除当前股票、空目录等场景不会产生多余 interval 或 loading 卡死
- 旧请求不会覆盖新股票状态，现有 `AbortController + requestId` 保护保持有效
- 相关实现完成后，至少通过 `lint` 与核心主路径自测

## 建议自测清单

- 交易时段内进入页面，确认自动轮询正常启动
- 将系统时间模拟到 `11:30` 前后，确认午休时自动停轮询
- 将系统时间模拟到 `13:00` 前后，确认下午自动恢复轮询
- 将系统时间模拟到 `15:00` 前后，确认收盘后自动停轮询
- 页面停留过夜或手动模拟到次日 `09:30`，确认自动恢复轮询
- 休市时手动点击“立即刷新”，确认只触发一次请求，不启动持续轮询
- 快速切换股票并观察网络请求，确认不存在旧响应覆盖与重复轮询
