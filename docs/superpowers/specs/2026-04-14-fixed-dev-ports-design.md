# 固定开发端口与启动前释放设计

## 背景

当前后端固定使用 `3000` 端口，但前端 `vite` 默认在 `5173` 被占用时会自动切换端口，导致开发地址不稳定。与此同时，后端在 `3000` 被占用时会直接启动失败，已经通过独立脚本处理了后端端口释放，但前端尚未纳入同一规则。

## 目标

- 前端固定使用 `5173` 端口。
- 后端固定使用 `3000` 端口。
- 每次启动前端或后端前，先自动释放对应端口占用。
- `npm run dev` 启动时，两个服务都走“先释放端口，再启动”的统一流程。

## 非目标

- 不修改业务接口、页面逻辑和轮询逻辑。
- 不引入新的端口配置中心或环境变量体系。
- 不处理除 `5173` 和 `3000` 之外的其它端口。

## 方案

### 固定端口

- 在 `vite.config.ts` 中显式设置前端开发端口为 `5173`。
- 开启 `strictPort: true`，确保前端不会静默切换到其它端口。
- 后端继续保留现有 `3000` 端口。

### 启动前释放

沿用现有的 `scripts/release-port.js`，在 `package.json` 中新增：

- `predev:frontend`: 启动前释放 `5173`
- `predev:backend`: 启动前释放 `3000`

这样无论执行 `npm run dev`、`npm run dev:frontend` 还是 `npm run dev:backend`，都会先执行对应的端口释放动作。

### 启动链路

- `npm run dev`
  - `concurrently` 并发启动前后端
  - `npm` 会先分别执行 `predev:frontend` 和 `predev:backend`
  - 端口释放完成后再启动服务

## 风险与稳定性分析

- 若前端仍允许自动换端口，会和“固定端口”的目标冲突，因此必须启用 `strictPort: true`。
- 启动脚本若误杀非目标端口进程，会影响其它项目；因此端口释放脚本必须只按目标端口查询 PID 并精确执行。
- 若释放脚本失败但继续启动，会出现端口冲突与地址不一致，因此释放失败应直接中断启动。

## 验收标准

- 执行 `npm run dev` 时，前端固定启动在 `http://localhost:5173`。
- 执行 `npm run dev` 或 `npm run dev:frontend` 时，前端启动前先处理 `5173`。
- 执行 `npm run dev` 或 `npm run dev:backend` 时，后端启动前先处理 `3000`。
- `npm run build` 和现有测试全部通过。
