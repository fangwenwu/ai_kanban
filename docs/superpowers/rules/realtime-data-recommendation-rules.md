# 实时数据与推荐门禁规则

## 适用范围

适用于本项目所有涉及以下能力的接口、分析逻辑和前端展示：

- 实时行情；
- 趋势分析；
- 买卖建议；
- 资金流展示；
- 自动刷新与轮询。

## 强制规则

### 1. 数据质量先于推荐输出

- 任何推荐结果都必须先经过统一的数据质量门禁；
- 禁止在未校验新鲜度、真实性、完整性、一致性的情况下直接输出买入 / 卖出建议；
- 质量分低于最低门槛时，只允许输出观望。

### 2. 实时数据必须标记新鲜度

- 关键实时数据必须带 `updatedAt`；
- 服务端必须计算与当前时间的秒级差值；
- 超过 15 秒的关键实时数据必须标记为过期；
- 过期数据不得输出高置信建议。

### 3. 关键字段必须可校验真实性

- 实时行情至少对 `price`、`changePercent`、`volume`、`updatedAt` 做双源校验；
- 校验失败时必须返回具体字段、主备值和阈值；
- 真实性未通过时，建议必须降级或阻断。

### 4. 估算值与真实值强制分离

- 估算值必须使用带 `estimated` 语义的独立字段；
- 禁止复用真实字段名承载估算值；
- 前端必须显式标注“估算”，不得以真实语义文案展示；
- 估算值只能作为辅助因子，不能作为高置信核心证据。

### 5. 推荐必须可解释

- 每条建议都必须返回：
  - 质量门禁状态；
  - 置信度原因；
  - 关键通过项；
  - 关键失败项；
  - 证据分拆解；
- 不允许只返回动作和一个笼统的“高 / 中 / 低”。

### 6. 横盘与冲突场景优先保守

- 横盘信号密集时，优先输出观望；
- 核心趋势证据不一致时，不允许依赖单个实时因子强推方向；
- 目标价与失效价方向不成立时，必须自动降级。

### 7. 异步链路必须竞态安全

- 每次分析请求必须有唯一 `requestId`；
- 新请求发起后旧请求响应不得覆盖最新状态；
- 自动刷新必须确保同一标的同时最多一个活跃请求；
- 组件卸载时必须清理轮询、监听和未完成请求。

## 高频问题与防复现措施

### 高频问题 1：估算资金冒充真实资金

修正路径：

- 拆分真实值字段与估算值字段；
- 增加 `sourceType`；
- 前端文案按来源分层展示。

防复现要求：

- Code Review 必查资金字段命名和来源标记；
- 测试必须覆盖“仅估算可得”场景；
- 若出现混用，直接驳回。

### 高频问题 2：未过质量门禁直接给强推荐

修正路径：

- 推荐逻辑统一从质量层入口进入；
- 输出结构增加 `qualityGate` 和 `degradeReasons`；
- 高置信建议必须绑定质量分门槛。

防复现要求：

- Code Review 必查推荐入口是否先过质量层；
- 测试必须覆盖过期、冲突、缺失三类降级场景；
- 若存在绕过门禁的快捷分支，直接驳回。

## 执行要求

- 开发阶段：新增或修改推荐逻辑时，必须先补质量门禁；
- 测试阶段：至少覆盖正常、过期、冲突、缺失、估算五类场景；
- 审查阶段：重点检查异步顺序、旧响应覆盖、重复监听、状态无法退出；
- 发布阶段：接口字段语义变化必须同步更新类型与前端展示。
